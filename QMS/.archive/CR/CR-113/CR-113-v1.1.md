### 5.4 Quality Manual: document `checkout from REVIEWED` reopen path

When a non-executable document is in `REVIEWED` (or an executable document is in `POST_REVIEWED`), and the responsible user wants to re-open it for further revision before routing for approval, the only valid action is `checkout`. The CLI silently auto-creates a new draft and returns the document to `DRAFT` (or `IN_EXECUTION`). `qms withdraw` rejects from these states because withdraw is defined only for transient routing states (`IN_REVIEW`, `IN_APPROVAL`, etc.).

This is a real and correct mechanism, but it is documented nowhere. This section makes three definite edits across three files.

#### 5.4.1 `Quality-Manual/guides/routing-quickref.md`

The "Non-Executable Documents" status table currently lists for `REVIEWED`:

```
| **REVIEWED** | `route --approval` | IN_APPROVAL |
```

Add a row immediately after, in the same table:

```
| **REVIEWED** | `checkout` | DRAFT (re-opens for revision before approval) |
```

The "Executable Documents" status table currently lists for `POST_REVIEWED`:

```
| **POST_REVIEWED** | `checkout` | IN_EXECUTION |
```

Replace this row's third column to match the wording of the new non-executable row:

```
| **POST_REVIEWED** | `checkout` | IN_EXECUTION (re-opens for revision before approval) |
```

#### 5.4.2 `Quality-Manual/03-Workflows.md`

The non-executable lifecycle "Transitions" table row currently reads:

```
| REVIEWED | DRAFT | (implied by rejection logic) | Owner | If any reviewer requested updates, owner must revise and re-route |
```

Replace with:

```
| REVIEWED | DRAFT | `checkout` | Owner | Re-opens for revision before approval routing. Auto-creates new draft (vN.X+1). Use this when reviewers requested updates that need addressing. |
```

#### 5.4.3 `Quality-Manual/guides/review-guide.md`

The §"When to Withdraw" subsection currently lists `IN_REVIEW`, `IN_APPROVAL`, `POST_REVIEW`, `POST_APPROVAL` as withdrawable states. After the existing table mapping "Withdrawn From → Returns To", add a follow-up paragraph:

```
**To re-open from REVIEWED or POST_REVIEWED:** `qms withdraw` is not valid from these states. Use `qms checkout {DOC_ID}` instead — this auto-creates a new draft revision. The document returns to DRAFT (or IN_EXECUTION) and is ready for further edits before routing again.
```

### 5.5 Quality Unit Handbook: strengthen "assign before review" rule

In `Quality-Manual/guides/quality-unit-handbook.md` §"The First Thing You Do", currently:

```
1. **Read the document.** All of it. ...
2. **Identify the document type** ...
3. **Determine which domains are affected** ...
4. **Assign reviewers** before starting your own review.

Assigning reviewers first is important because it lets them work in parallel with you.
```

Replace step 4 and the trailing rationale with:

```
4. **Assign reviewers BEFORE submitting your own review.**

Assigning first is required, not just preferred. The CLI's automatic transition from IN_REVIEW (or IN_PRE_REVIEW / IN_POST_REVIEW) to REVIEWED triggers as soon as all assigned reviewers have responded. If you are the sole assigned reviewer at the moment you submit your outcome, the cycle closes immediately, and the `assign` command becomes invalid (it requires IN_REVIEW or IN_APPROVAL). Adding TUs after this point requires the document owner to withdraw and re-route, which restarts the review cycle.

Assigning first also lets reviewers work in parallel with you, which is the secondary benefit.
```

### 5.6 Quality Unit Handbook: re-review baseline guidance

In `Quality-Manual/guides/quality-unit-handbook.md` §"Judgment Calls Only QA Can Make", add a new subsection after the existing "When to Challenge the VAR Type" subsection:

```
### Re-Review Baseline After Withdrawal

When a draft has been withdrawn from review and re-routed for another cycle (whether due to your own request-updates, an orchestrator-initiated revision, or a procedural reset), the relevant comparison baseline for "did the substantive content change?" is **the last EFFECTIVE version of the document, not the prior review cycle's content**.

This matters most when you are deciding whether previously-assigned reviewers need to re-review the new draft. If the prior cycle's reviewers had not yet submitted their reviews when the document was withdrawn, they have not actually reviewed any content — recent or otherwise. The "previous recommendation" you might be tempted to lean on belongs to an earlier *effective* version, not to the current draft cycle.

**Rule:** When deciding whether re-review by domain TUs is required, compare the current draft's substantive content (evidence text, technical claims, requirement statements) to the last EFFECTIVE version. If those differ in substance — not merely in line citations or formatting — domain TUs must review. The fact that QA's own concerns from a prior cycle have been addressed is necessary but not sufficient.
```

```
## Invocation Modes

You may be invoked in either of two modes. Both deliver the same QMS work; the difference is in how the work reaches you.

### Task tool subagent (one-shot)

The orchestrator (`claude`) spawns you via Claude Code's Task tool with a starting prompt. The prompt typically identifies a document and a task type (review, approval). On invocation:

1. Run `qms_inbox(user="{IDENTITY}")` (or the CLI equivalent) to see your pending tasks.
2. Process each task per your role and the SOPs.
3. Submit your outcome via the QMS CLI / MCP tools.

The orchestrator may resume you in subsequent calls within the same session for additional tasks.

### Containerised agent (long-running)

You run as a persistent process in an agent-hub container. Task notifications arrive typed directly into your input prompt via tmux send-keys, in the form:

```
Task notification: CR-058 review is in your inbox. Please run qms_inbox() to see your pending tasks.
```

When you receive a notification:

1. Run `qms_inbox(user="{IDENTITY}")` to see your pending tasks.
2. Process each task according to your role and the SOPs.
3. When finished, return to idle and await further notifications.

If you are mid-task when a notification arrives, it queues and appears at your next prompt. Process it when you are ready.

### What is the same in both modes

Your authoritative source of work is your QMS inbox, not the orchestrator's prompt or any tmux notification. Both invocation paths exist only to *route* you to the inbox; the inbox is what tells you which document is yours, what task type, and at what version.

Your behaviour, role, permissions, and review criteria are identical across modes.
```

Substitute `{IDENTITY}` with the agent's identity (`qa`, `tu_ui`, etc.) when applying to each file.

This change replaces the existing "Inbox Notifications" section entirely. The container-mode information is preserved in the second subsection; the new content is the Task-tool subsection and the closing paragraph that explicitly grounds both modes in the inbox as the source of truth (reinforcing SOP-007 §5).

## 6. Justification

- **Friction encountered in CR-112 EI-10 (RTM v1.1 cycle).** Took four review cycles to land instead of two because of two procedural gaps (CLI ordering, re-review baseline) and one undocumented mechanism (REVIEWED → DRAFT). Documenting these now reduces the cost the next time the situation arises.
- **Stale references mislead reviewers.** A TU agent reading its own definition file is told to consult a Design Specification that does not and will not exist (per SOP-006 §4) — this is a direct contradiction with the SOP framework. The "one-way bridge" framing in tu_sketch and tu_sim contradicts CR-112's qualified state of the codebase. Both are sources of low-grade confusion every time those agents are spawned.
- **The Quality Manual is meant to be a portable framework.** Per the Lead's explicit direction, it should remain generic about reviewer naming. But the project still needs a place where QA can find "the Pipe Dream TUs are X, Y, Z, and they cover these directories." The QA agent definition file is the natural place: it is project-specific, agent-specific, and read by QA on every invocation.
- **The orchestrator's SOP-007 §5.3 violations were systematic, not incidental.** CR-112's TU-spawning prompts repeatedly included summaries, verification checklists, and architectural sub-questions. The reviewers ultimately formed independent judgments, but review independence is structural — cooperative violations don't vindicate the framing. A brief CLAUDE.md reminder makes the constraint visible at the moment of agent spawning, where SOP-007 itself is too far away to consult naturally.
- **Both invocation paths are in active use.** The agent-hub container mode and the Task tool subagent mode have both been used for QMS reviews. Documenting only one creates a small but recurring source of confusion: an agent reading its own definition file under the Task tool sees only container-mode guidance and may wonder whether tmux notifications are coming. Documenting both paths and grounding them in the inbox-as-source-of-truth principle removes that confusion at near-zero cost.
