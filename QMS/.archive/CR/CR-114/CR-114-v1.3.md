# CR-114: Resize World UX fixes and brush_tool orphan deletion

## 1. Purpose

Three items in scope:

1. The Resize World input field is constructed at zero width and zero height, making it invisible — the button is wired to read from a control the user cannot see or edit.
2. Clicking the Resize World button silently wipes all simulation particles with no confirmation.
3. `flow-state/ui/brush_tool.py` is dead code superseded by the `BrushTool` in `ui/tools.py:124` (CR-112 ToolContext migration completion); it was identified as removable during CR-112 scoping.

Items 1 and 2 are latent defects in the Resize World feature. Item 3 is an independent cleanup queued during CR-112 scoping (Session-2026-04-30-002), bundled here for execution efficiency.

This is the first CR after CR-112 that exercises the "lightweight CR with no RS revision" pattern — bug fixes within the existing RS surface, plus a queued cleanup, all in one narrow scope.

## 3. Current State

- The Resize World button (`ui_manager.py:291`, registered as `buttons['resize']`) is visible in the editor right panel.
- Its paired input field is constructed in `flow_state_app.py:109` as `InputField(0, 0, 0, 0)` with no initial value, then attached to `Session.input_world` so its value persists across UI rebuilds. UIManager treats this truthy zero-sized field as the supplied input and never falls into the proper-dimensions fallback at `ui_manager.py:286`.
- Clicking the Resize World button calls `controller.actions.action_resize_world(value)`, which calls `sim.resize_world(val)`. That method calls `self.clear(snapshot=False)` after setting `world_size`, silently destroying all particles. The `snapshot()` call at the start uses the simulation's own particle-undo mechanism (used by Brush operations), not the Command queue, so Ctrl+Z does not restore the wiped particles.
- `flow-state/ui/brush_tool.py` (~145 lines) defines a standalone BrushTool that uses lazy-initialized ParticleBrush and direct app references — superseded by the ToolContext-based BrushTool in `ui/tools.py:124-242` (CR-112). No imports of `ui.brush_tool` exist in the active codebase.

The Session-attached field pattern is preserved — the field continues to live on `Session.input_world` so its current value survives UI rebuilds on window resize and mode switch (per `flow_state_app.py:127, 232, 334`).

### 5.2 Confirmation dialog widget (ui_widgets.py)

Add a small generic `ConfirmDialog` class to `ui/ui_widgets.py`, modeled on the existing dialog classes (`SaveAsNewDialog`, `MaterialDialog`, `RotationDialog`, `AnimationDialog`) and following the established **polled-flag pattern** the codebase uses for modal dialog completion:

**Constructor:**

- `title: str` — dialog title
- `message: str` — body message (newlines supported)
- `destructive: bool = False` — when True, Cancel becomes the default keyboard target

**Internal state (polled by the dispatcher):**

- `done: bool` — set to True when the user has made a choice or the dialog is dismissed
- `confirmed: bool` — set to True only when the user explicitly clicked Confirm; False on Cancel, click-outside, or Escape
- `cancelled: bool` — set to True for any non-confirm completion (Cancel button, click-outside, Escape key); useful symmetry with `SaveAsNewDialog.cancelled`

**Interface (matches existing dialog classes):**

- `handle_event(event) -> bool` — process pygame events; sets `done`/`confirmed`/`cancelled` on completion
- `draw(screen, font)` — render via the existing `actions.draw_overlays(screen, font)` dispatch (per `app_controller.py:598-603`)
- `update(dt)` — no-op; satisfies the `actions.update()` modal-iteration loop at `app_controller.py:585-596`

**Keyboard handling (explicit, load-bearing for the destructive-default UX):**

- **Enter** when `destructive=False` → triggers Confirm (sets `confirmed=True, done=True`)
- **Enter** when `destructive=True` → triggers Cancel (sets `cancelled=True, done=True`) — a hasty Enter never fires a destructive action
- **Escape** (always) → triggers Cancel
- **Mouse click on Confirm button** → triggers Confirm
- **Mouse click on Cancel button** → triggers Cancel
- **Click outside the modal panel** → triggers Cancel (handled by the dispatcher in `input_handler.py`, see §5.3)

**Visual focus:**

- When `destructive=True`, the Cancel button receives the highlight ring / default-focus visual treatment so the user sees at a glance that Enter is the safe path.

Generic by design — though only one caller is added in this CR, the dialog is intended to be reusable for any future destructive action that warrants confirmation. The `destructive` flag is the standard UX pattern for distinguishing benign confirmations (Enter = Confirm) from destructive ones (Enter = Cancel).

### 5.3 Confirmation flow (app_controller.py + input_handler.py)

Modify `AppController.action_resize_world()` to push a `ConfirmDialog` onto the modal stack with the resize value stashed on the dialog instance, and add two new helpers:

```python
# Before (current behavior)
def action_resize_world(self, size_str):
    try:
        val = float(size_str)
        self.sim.resize_world(val)
        self.session.status.set(f"World Resized: {val}")
        self.sound_manager.play_sound('click')
    except ValueError:
        self.session.status.set("Invalid Size")

# After
def action_resize_world(self, size_str):
    try:
        val = float(size_str)
    except ValueError:
        self.session.status.set("Invalid Size")
        return

    if self.sim.count > 0:
        # Particles present — confirm before destructive reset
        dialog = ConfirmDialog(
            title="Resize World",
            message=f"Resize the world to {val} and reset the simulation?\n"
                    f"All {self.sim.count} particles will be cleared and physics "
                    f"settings will return to defaults.",
            destructive=True,
        )
        dialog.pending_value = val   # stash for the dispatcher
        self.push_modal(dialog, 'confirm_resize_dialog')
    else:
        self._do_resize_world(val)

def _do_resize_world(self, val):
    self.sim.resize_world(val)
    self.scene.rebuild()    # restore Compiler-emitted static/tethered atoms wiped by reset()
    self.session.status.set(f"World Resized: {val}")
    self.sound_manager.play_sound('click')

def apply_resize_confirm(self, dialog):
    """Dispatcher hook for 'confirm_resize_dialog' completion."""
    if dialog.confirmed:
        self._do_resize_world(dialog.pending_value)
    else:
        self.session.status.set("Resize cancelled")
    self.close_modal(dialog)
```

**`scene.rebuild()` rationale:** `Simulation.reset()` clears all particles including the static atoms (`is_static=1`) and tethered atoms (`is_static=3`) that the Compiler emitted from CAD geometry. Without an explicit rebuild, post-resize the simulation has no atoms representing the still-present CAD walls — collisions silently break and the two-way CAD↔Physics coupling breaks until something else dirties topology. Calling `self.scene.rebuild()` regenerates those atoms immediately, mirroring the precedent at `action_clear_particles` (`app_controller.py:174-178`).

**Dispatcher branch (input_handler.py):** Add a `'confirm_resize_dialog'` branch in `_attempt_handle_modals()` (currently `flow-state/ui/input_handler.py:225-283`):

```python
# In the click-outside-to-dismiss section (around line 250):
if not click_inside:
    if modal_type == 'save_as_new_dialog':
        modal.cancelled = True
        modal.done = True
    elif modal_type == 'confirm_resize_dialog':
        modal.cancelled = True
        modal.done = True
    else:
        actions.close_modal()
    return True

# In the modal completion section (around line 273, after anim_dialog):
elif modal_type == 'confirm_resize_dialog':
    if getattr(modal, 'done', False):
        actions.apply_resize_confirm(modal)
```

The polled-flag pattern is consistent with `prop_dialog`, `anim_dialog`, and (for click-outside dismissal) `save_as_new_dialog` — no new dialog idiom is introduced.

The dialog uses the existing `push_modal` infrastructure. On dismissal, the modal-stack `reset_interaction_state()` cascade (per CLAUDE.md §6.4) prevents ghost-clicks on the Resize World button itself.

### 5.4 Full physics reset (simulation.py)

Modify `Simulation.resize_world()` at `flow-state/engine/simulation.py:269`:

```python
# Before
def resize_world(self, new_size):
    """Resize the simulation world."""
    self.snapshot()
    if new_size < 10.0:
        new_size = 10.0
    self.world_size = new_size
    self.clear(snapshot=False)

# After
def resize_world(self, new_size):
    """Resize the simulation world. Performs a full physics reset.

    All particles are cleared and physics settings (gravity, dt, damping,
    sigma, epsilon, skin_distance, target_temp) return to defaults.
    The new world_size overrides the default.
    """
    if new_size < 10.0:
        new_size = 10.0
    self.reset()                  # full physics reset; reset() snapshots internally
    self.world_size = new_size    # override the default set by reset()
```

The order matters: `reset()` first (which sets `world_size = DEFAULT_WORLD_SIZE` and takes a snapshot internally — see `simulation.py:292`), then assign the user's chosen `new_size`. The outer `snapshot()` call from the prior shape is **removed** — `reset()` already snapshots, so retaining the outer call would push two undo states per resize and require two Ctrl+Z presses to revert.

## 6. Justification

**Defect repair:** The Resize World feature is shipped but materially broken. The user cannot edit the world size through the intended path because the input field is invisible. Even when the button is clicked (using the field's hardcoded zero-width placeholder value), the destructive side effect is undocumented in the UI.

**User protection:** The destructive `clear()` happens silently. A user who has built up a non-trivial particle simulation and accidentally clicks Resize World loses their work with no opportunity to abort. A confirmation dialog adds a single clarifying click; it does not impede the rare-and-deliberate resize operation.

**Mental-model alignment:** "Resize the world" semantically implies a fresh canvas. The current behavior wipes particles but leaves stale physics tuning (gravity, damping, etc.) — an unprincipled middle state. Full reset is a cleaner contract: "I am starting over with a different-sized world."

**Code hygiene:** Orphan files create archaeology overhead. `ui/brush_tool.py` was superseded by the ToolContext-based `BrushTool` in `ui/tools.py` during the CR-112 ToolContext migration completion; it has been dead since. Removing it eliminates a confusion vector for future contributors.
