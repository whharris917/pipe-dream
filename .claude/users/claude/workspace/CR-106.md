# CR-106: System Governance: get / checkout / checkin / lock / permit for governed submodules

## 1. Purpose

Extend the QMS CLI to treat governed submodules ("systems") as first-class entities with registration, exclusive locking, qualification-gated permits, and versioned releases. This is the foundational step toward the Guided Agent Orchestration vision: agents cannot modify a governed system without an authorizing CR, an exclusive lock, and a qualification-gated permit — all enforced programmatically by the CLI.

---

### 2.1 Context

Currently, execution branches, merge gates, and system releases are managed by convention (CLAUDE.md instructions, SOP-005 prose, write-guard hooks). This CR makes the CLI the enforcement mechanism.

## 3. Current State

The QMS CLI manages documents through their full lifecycle (create, checkout, checkin, route, review, approve) but has no awareness of governed submodules. Execution branches, merge gates, system releases, and write permissions to submodules are enforced by convention and hooks, not by the CLI. SDLC namespaces (QMS, CQ, FLOW) exist but only generate RS/RTM document types — they have no concept of the underlying system they govern.

---

## 4. Proposed State

Governed submodules are registered as "systems" in the QMS. Each system has metadata tracking its release number, qualified commit, lock state, and permit state. Agents use `qms checkout --system` to acquire exclusive locks (requiring an authorizing CR), `qms permit --system` to request qualification evaluation, and `qms checkin --system` to release with a new version number. The CLI refuses checkin without a granted permit, preventing unqualified code from reaching main.

---

## 5. Change Description

### 5.1 System Registry

Systems are registered as metadata sidecar files at `QMS/.meta/SYSTEM/{system-id}.json`, following the same pattern as document metadata. Each system record includes:

- Identity: `system_id`, `submodule_path`, `sdlc_namespace`
- Versioning: `current_release` (simple integer), `qualified_commit`
- Lock state: `locked`, `locked_by`, `authorizing_cr`, `lock_date`, `execution_branch`, `base_commit`
- Permit state: `permit_granted`, `permit_conditions` (RS/RTM versions and status, tests_pass flag)

### 5.2 Command: `qms system register NAME`

Administrator-only. Creates the system metadata file. Validates that the submodule path exists and the SDLC namespace is registered. Arguments: `--path`, `--namespace`, `--release`, `--commit`.

### 5.3 Command: `qms system list` / `qms system status NAME`

Query commands. `list` shows all registered systems with current state. `status` shows detailed info for one system including lock holder, permit conditions, and associated RS/RTM versions.

### 5.4 Command: `qms get --system NAME`

Clones/references the system for read-only exploration. No lock, no exclusivity. Writes a `.ref.json` file to the user's workspace with the system's current commit and release. Use case: reviewers examining code, container-based agents needing workspace copies.

### 5.5 Command: `qms checkout --system NAME --cr CR-NNN`

Acquires an exclusive lock on the system. Preconditions: system not locked by another user, CR is IN_EXECUTION, CR declares this system in `target_systems`, user is the CR's responsible_user. Updates system metadata (locked=true, etc.) and writes a lock file to workspace. Does NOT create the git branch — that remains agent work.

### 5.6 Command: `qms permit --system NAME`

Evaluates qualification conditions: RS EFFECTIVE + RTM EFFECTIVE + tests pass (via `--tests-pass` flag). If all conditions met, sets `permit_granted=true` and records the RS/RTM versions in `permit_conditions`. If any condition fails, reports which are unmet. Permit auto-revokes if RS/RTM versions no longer match (checked lazily at `permit` and `checkin` time).

### 5.7 Command: `qms checkin --system NAME`

Requires permit. Re-verifies permit conditions are still valid. Increments `current_release`, updates `qualified_commit`, clears lock and permit state, removes workspace lock file. Does NOT merge the git branch — that remains agent work.

### 5.8 CR Metadata Extension

Add `target_systems` field (list of strings) to CR metadata via `create_initial_meta()`. Populated during CR authoring. Validated during system checkout.

---

## 6. Justification

- Currently, nothing prevents an agent from modifying a governed system without authorization — enforcement relies on conventions and hooks that can be bypassed
- The permit system makes the merge gate programmatic: agents physically cannot check in a system until RS, RTM, and tests are all qualified
- The lock system prevents concurrent modification of the same system by multiple CRs
- This is the foundational infrastructure for the Guided Agent Orchestration vision, where the CLI enforces compliance programmatically rather than relying on agents to read and internalize procedures

- System metadata CRUD operations (create, read, update, write)
- Register command: success, duplicate rejection, invalid namespace, invalid path
- Get command: ref file creation, no lock acquired
- Checkout command: lock acquisition, concurrent lock rejection, CR validation (not in-execution, wrong target_systems, wrong user)
- Permit command: all conditions met → granted, individual condition failures, auto-revocation on RS/RTM change
- Checkin command: permit required, re-validation at checkin time, release increment, cleanup
- Permission checks: admin-only for register, owner-only for checkin/permit
- Full regression suite (existing 687 tests unaffected)

### Integration Verification

- End-to-end workflow: register system → create CR with target_systems → release CR → checkout system → (simulate qualification) → permit → checkin → verify release incremented
- Concurrent lock rejection: two users attempt checkout
- Permit revocation: grant permit, then checkout RS (breaking EFFECTIVE status), verify permit revoked
