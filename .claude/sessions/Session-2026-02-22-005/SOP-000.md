# SOP-000: Quality Management System

---

## 1. Purpose

This document defines the complete policy specification for the Flow State Quality Management System (QMS). It governs all document control, change control, deviation management, execution, code governance, SDLC processes, and agent orchestration.

The QMS is a recursive governance system: it controls how the Flow State application is developed, and it controls its own evolution through the same mechanisms. Process failures become inputs to process improvement via the same document control infrastructure.

---

## 2. Scope

This SOP applies to all controlled documents, all authorized users (human and AI), and all repositories governed by the QMS.

## 4. Governance Model

### 4.1 User Groups

| Group | Role |
|-------|------|
| **Administrator** | All Initiator permissions plus administrative commands |
| **Initiator** | Create and manage documents through workflow |
| **Quality** | Assign reviewers; review and approve documents |
| **Reviewer** | Review and approve assigned documents |

### 4.2 Permission Matrix

| Action | Administrator | Initiator | Quality | Reviewer |
|--------|:---:|:---:|:---:|:---:|
| create, checkout, checkin | Yes | Yes | - | - |
| route, release, revert, close | Yes | Yes | - | - |
| assign | - | - | Yes | - |
| review | When assigned | When assigned | When assigned | When assigned |
| approve, reject | - | - | When assigned | When assigned |
| fix | Yes | - | - | - |

### 4.3 Agent Orchestration

The QMS uses a hub-and-spoke model. A single **Orchestrator** (claude) coordinates all work and communicates with the Lead. **Subagents** (qa, tu_ui, tu_scene, tu_sketch, tu_sim, bu) provide domain expertise via the review/approval workflow.

**Communication boundaries:**
- Agents communicate only through QMS mechanisms (reviews, comments, inbox tasks).
- Subagents shall not receive full conversation context from the orchestrator.
- Each agent reasons independently based on the document content it is assigned to review.

**Review independence:**
- Reviewers form opinions based solely on the document under review and applicable SOPs.
- The orchestrator shall not coach, pre-empt, or influence reviewer conclusions.
- Disagreements between reviewers are resolved by escalation to the Lead, not by the orchestrator.

## 5. Document System

### 5.1 Document Types

| Type | Executable | Parent Required | Parent State Constraint | Notes |
|------|:---:|:---:|:---:|-------|
| SOP | No | - | - | Standalone procedure |
| CR | Yes | - | - | Change authorization |
| INV | Yes | - | - | Deviation investigation |
| TP | Yes | CR | - | Test protocol; singleton child of CR |
| ER | Yes | TP | - | Exception report; child of TP |
| VAR | Yes | CR, INV, TP, VAR | None | Variance during execution |
| ADD | Yes | CR, INV, VAR, ADD | CLOSED | Post-closure correction |
| VR | Yes | CR, VAR, ADD | IN_EXECUTION | Verification record; born IN_EXECUTION |
| RS | No | - | - | Requirements specification; singleton per SDLC namespace |
| RTM | No | - | - | Requirements traceability matrix; singleton per SDLC namespace |

### 5.2 Document Naming

| Type | Pattern | Example |
|------|---------|---------|
| SOP, CR, INV | `{TYPE}-NNN` | CR-005 |
| TP | `{PARENT}-TP` | CR-005-TP |
| ER, VAR, ADD, VR | `{PARENT}-{TYPE}-NNN` | CR-005-VAR-001 |
| RS, RTM | `SDLC-{NAMESPACE}-{TYPE}` | SDLC-FLOW-RS |
| Template | `TEMPLATE-{NAME}` | TEMPLATE-CR |

Child documents nest: `CR-005-VAR-001-ADD-001`. Sequential numbers are zero-padded to 3 digits.

File states are distinguished by suffix:
- Effective: `{DOC_ID}.md`
- Draft: `{DOC_ID}-draft.md`
- Archived: `{DOC_ID}-v{N.X}.md`

### 5.3 Metadata Architecture

Document metadata is separated into three tiers:

| Tier | Contents | Managed By |
|------|----------|------------|
| Frontmatter | Title, revision summary | Author |
| `.meta/` sidecar | Version, status, responsible user, checkout state, assignees | CLI exclusively |
| `.audit/` log | Append-only chronological event history | CLI exclusively |

**Frontmatter policy:**
- `revision_summary` should reference the authorizing CR ID when the revision is driven by a CR.
- Exceptions: initial creation (v0.1), self-authorizing executable documents, revisions not driven by a CR.

**Content standards:** Documents shall not contain version numbers, effective dates, responsible user identifiers, or revision history tables in their body. All such metadata is available through the CLI.

### 5.4 Versioning

Format: `N.X` where N = approval count (major), X = revision within cycle (minor).

| Event | Version |
|-------|---------|
| Create | 0.1 |
| Checkout from effective vN.0 | N.1 |
| Approval | (N+1).0 |

The effective version of a document is always a major version (N.0). Minor versions exist only as drafts.

## 6. Lifecycle

### 6.1 Non-Executable Lifecycle

```
DRAFT ──► IN_REVIEW ──► REVIEWED ──► IN_APPROVAL ──► EFFECTIVE
```

On approval, documents transition through APPROVED to EFFECTIVE automatically. The previous effective version (if any) is archived. The new draft is promoted to the effective path.

### 6.2 Executable Lifecycle

```
DRAFT
  ──► IN_PRE_REVIEW ──► PRE_REVIEWED
  ──► IN_PRE_APPROVAL ──► PRE_APPROVED
  ──► IN_EXECUTION
  ──► IN_POST_REVIEW ──► POST_REVIEWED
  ──► IN_POST_APPROVAL ──► POST_APPROVED
  ──► CLOSED
```

Executable documents have two approval gates (pre-execution and post-execution) with an execution phase between them.

### 6.3 VR Lifecycle

VR documents skip the entire pre-approval cycle. They are created directly in IN_EXECUTION at version 1.0. Their lifecycle begins at execution and proceeds through post-review, post-approval, and closure.

A parent document must be IN_EXECUTION to create a VR child.

### 6.4 Terminal States

| State | Applies To | Recovery |
|-------|------------|----------|
| CLOSED | Executable | No (but ADD can supplement a closed document) |
| RETIRED | All | No |

### 6.5 Withdrawal

The document owner may withdraw from any active workflow state:

| From | Returns To |
|------|------------|
| IN_REVIEW / IN_PRE_REVIEW | DRAFT |
| IN_APPROVAL / IN_PRE_APPROVAL | REVIEWED / PRE_REVIEWED |
| IN_POST_REVIEW | IN_EXECUTION |
| IN_POST_APPROVAL | POST_REVIEWED |

Checkout by the owner from an active workflow state performs an implicit withdrawal.

## 7. Workflow Rules

### 7.1 Check-Out and Check-In

- A document must be checked out before editing. Checkout locks the document to a single responsible user.
- When an effective document is checked out, the effective version remains in force while the draft is developed (concurrent effective + draft).
- Only the responsible user may check in a document.
- Checking in from a REVIEWED or PRE_REVIEWED state reverts the document to DRAFT (new content requires re-review).

### 7.2 Review

- Reviews must precede all approvals. Multiple review rounds are permitted.
- QA is automatically assigned to all routed documents.
- QA assigns additional reviewers (Technical Units) based on affected domains.
- Review outcomes: **recommend** (proceed) or **request-updates** (revision required).
- When all assigned reviewers complete, the document transitions automatically to its reviewed state.
- The CLI infers pre-review vs post-review from the document's current status.

### 7.3 Approval

- Approval requires all of the following (approval gate):
  1. All assigned reviewers have recommended.
  2. At least one quality-group member has reviewed.
- If any approver rejects, the document returns to its previous reviewed state. The responsible user must address rejection comments and route for a full review cycle before re-attempting approval.

### 7.4 Release and Close

- **Release:** Transitions from PRE_APPROVED to IN_EXECUTION. Must be explicit.
- **Close:** Transitions from POST_APPROVED to CLOSED. Must be explicit.
- **Cascade close:** When a parent document closes, any child VR documents that are not already in a terminal state are automatically closed.

### 7.5 Revert

A document in POST_REVIEWED state may be reverted to IN_EXECUTION if additional execution work is needed. A reason must be provided.

### 7.6 Cancel and Retire

- **Cancel:** Permanently deletes a document that has never been approved (version < 1.0). The document must not be checked out. All metadata and audit history are deleted.
- **Retire:** Documents that have been effective cannot be deleted. They are archived through a formal approval process using the retire flag, which transitions to the RETIRED terminal state instead of EFFECTIVE.

### 7.7 Administrative Fix

Administrators may perform corrective fixes on EFFECTIVE or CLOSED documents to clear stale metadata (e.g., orphaned checkout flags, missing effective dates).

## 8. Execution Policy

### 8.1 The Executable Block

Executable documents are divided into two blocks:

| Block | Sections | Timing |
|-------|----------|--------|
| Static | Planning, scope, methodology | Written pre-approval; locked after pre-approval |
| Editable | Execution tables, results, summaries | Completed during execution |

Pre-approval reviews verify the plan. Post-approval reviews verify the execution.

### 8.2 Execution Items (EI)

Work during execution is organized into an EI table:

| Column | Description |
|--------|-------------|
| EI | Sequential identifier (EI-1, EI-2, ...) |
| Task Description | What will be done (static, set pre-approval) |
| VR | Whether behavioral verification is required (Yes/No) |
| Execution Summary | What was actually done (editable during execution) |
| Task Outcome | Pass or Fail |
| Performed By -- Date | Who did it and when |

If VR = Yes, a VR child document must be created and executed before the EI can be marked complete.

### 8.3 Placeholder Convention

| Syntax | Replaced During | Example |
|--------|----------------|---------|
| `{{DOUBLE_CURLY}}` | Authoring (pre-approval) | `{{PARENT_DOC_ID}}` |
| `[SQUARE_BRACKETS]` | Execution (post-release) | `[RESOLUTION_DETAILS]` |

No `{{}}` placeholders shall remain when routing for pre-approval. No `[]` placeholders shall remain when routing for post-review.

### 8.4 Evidence Requirements

- Each EI must record sufficient evidence to demonstrate that the described task was completed.
- Evidence may include: commit hashes, test results, configuration screenshots, document references.
- Evidence must be recorded contemporaneously with execution, not reconstructed after the fact.

### 8.5 Scope Integrity

The scope of execution must match the scope of the pre-approved plan. If execution reveals that additional work is needed beyond the approved scope, that work must be captured in a child document (VAR or ADD), not silently added to the parent.

## 9. Child Documents

### 9.1 Variance Report (VAR)

A VAR is created when execution of a parent document cannot proceed as planned. VARs encapsulate resolution work in an independent review/approval cycle.

**Type classification:**

| Property | Type 1 | Type 2 |
|----------|--------|--------|
| Block condition | Full VAR closure required | VAR pre-approval sufficient |
| Use when | Resolution is critical to parent objectives | Variance is understood and contained |
| Parent can close when | VAR is CLOSED | VAR is PRE_APPROVED or later |

**Creation:** No parent status restriction. The parent document's EI table references the VAR ID but does not duplicate resolution details.

### 9.2 Addendum Report (ADD)

An ADD is created to correct or supplement a parent document that has already closed.

**Creation constraint:** Parent must be CLOSED. This is the defining distinction from VAR (which operates during execution).

**Scope handoff:** ADDs contain a mandatory scope handoff section confirming: what the parent accomplished, what the ADD corrects, and that no scope items were lost in the transition.

### 9.3 Verification Record (VR)

A VR is a pre-approved evidence form for structured behavioral verification. VRs use interactive authoring (see Section 10).

**Lifecycle:** Born IN_EXECUTION at v1.0. No pre-approval cycle — the VR template and its parent's pre-approval serve as the authorization.

**Parent constraint:** Parent must be IN_EXECUTION.

**Cascade close:** VRs are automatically closed when their parent document closes, including auto-compilation of interactive VRs still in progress.

### 9.4 Test Protocol (TP) and Exception Report (ER)

**TP:** Singleton child of a CR. Contains a scope table mapping test areas to verification methods, followed by embedded Test Cases (TC sections). TPs follow the standard executable lifecycle.

**ER:** Child of a TP, created when a test step fails during TP execution. Contains a re-test section using the same test script structure as TC. If the re-test also fails, a nested ER may be created.

## 10. Interactive Authoring

Interactive documents (VR) use a three-layer file model:

| Layer | File | Purpose |
|-------|------|---------|
| Template | `.interact` | Prompt flow definition using template tags |
| Data | `.source.json` | Accumulated responses (append-only during execution) |
| Rendered | `.md` | Compiled markdown document (generated from template + data) |

**Template tags** define the prompt flow: sequential prompts, gates (yes/no decisions that control flow), and loops (repeatable prompt groups). The engine presents prompts one at a time, records responses to the source data file, and compiles the rendered document after each response.

**Source data integrity:** The `.source.json` file is append-only during execution. Responses are never modified or deleted. Each response triggers an auto-commit for evidence capture.

**Compilation:** The rendered markdown is deterministically generated from the template and source data. It can be regenerated at any time. The source data is the authoritative record.

## 11. Change Control

### 11.1 CR Structure

Change Records authorize modifications to controlled systems. A CR contains:

1. **Change Identification** — what is being changed and why
2. **Technical Description** — detailed description of the change
3. **Impact Assessment** — what is affected
4. **Implementation Plan** — EI table with task descriptions
5. **Execution** — completed EI table with results (editable during execution)
6. **Execution Summary** — overall outcome narrative (editable during execution)

### 11.2 Review Team

QA assigns Technical Unit reviewers based on affected domains:

| Domain | Reviewer |
|--------|----------|
| UI, input handling, tools, widgets | tu_ui |
| Scene orchestration, commands, undo/redo | tu_scene |
| Sketch, geometry, constraints, solver | tu_sketch |
| Particle engine, simulation, Numba kernels | tu_sim |
| User experience, product value | bu |

### 11.3 Post-Review Requirements

Post-execution review must verify:
- All EIs are complete with recorded outcomes.
- All VR-flagged items have completed VR documents.
- All child documents (VAR, ADD) have met their block conditions.
- For code CRs: the execution branch has been merged to main via pull request, and the merge commit is reachable from HEAD.

### 11.4 Rollback

If a change causes unforeseen problems after closure, a new CR must be created to revert or correct the change. Rollback is itself a controlled change.

## 12. Deviation Management

### 12.1 When to Investigate

An Investigation (INV) shall be created when:
- A process failure occurs (procedure not followed, or procedure followed but produced wrong result)
- A systemic issue is discovered that may affect multiple documents or processes
- Root cause analysis is needed beyond what a simple VAR can provide

An INV is not required for:
- Isolated execution variances handled by a VAR
- Clarifications or enhancements captured in a CR

### 12.2 CAPA

Corrective and Preventive Actions are captured as EIs within the INV execution table:

| CAPA Type | Purpose |
|-----------|---------|
| Corrective | Fix the immediate problem |
| Preventive | Prevent recurrence |

Each CAPA-EI identifies: the action, the target document or process, and verification criteria. CAPAs that require document changes generate child CRs.

### 12.3 Closure Criteria

An INV may be closed when:
- Root cause has been identified
- All CAPA-EIs are complete
- All child CRs generated by CAPAs have reached their required state
- Effectiveness of corrective actions has been verified

## 13. Code Governance

### 13.1 Execution Branch Workflow

Code changes authorized by a CR are developed on an **execution branch** named after the CR (e.g., `cr-056-feature-name`). The main branch is protected; all changes reach main via pull request.

### 13.2 Qualified Commit

A **qualified commit** is a specific commit on the execution branch that has passed CI verification. The qualified commit hash is recorded in the RTM as verification evidence.

Requirements:
- The qualified commit must be on the execution branch, not main.
- CI must pass at that commit.
- The qualified commit must be reachable on main via the merge commit after PR merge.

### 13.3 Merge Gate

A PR from an execution branch to main requires:
1. RS is EFFECTIVE (requirements approved)
2. RTM is EFFECTIVE (verification evidence approved)
3. CI passes on the execution branch
4. CR post-review verifies the merge (PR URL recorded in execution evidence)

### 13.4 Genesis Sandbox

Initial project bootstrapping (before the QMS is operational) may use a **genesis sandbox** — a time-boxed period where code is developed without full CR governance. Genesis work is adopted into the QMS via an **adoption CR** that retroactively establishes traceability.

### 13.5 System Release Versioning

System releases follow semantic versioning. Each release is tagged and traceable to the set of CRs that comprise it.

## 14. SDLC Governance

### 14.1 Two-Document Framework

Each controlled system has two SDLC documents within a namespace:

| Document | Purpose |
|----------|---------|
| RS (Requirements Specification) | Defines what must be built |
| RTM (Requirements Traceability Matrix) | Proves it was built and verified |

RS and RTM are singleton documents per namespace (e.g., SDLC-FLOW-RS, SDLC-FLOW-RTM). Both follow the non-executable lifecycle.

### 14.2 Requirement Format

Requirements use the format `REQ-{NAMESPACE}-NNN` (e.g., `REQ-FLOW-001`). Each requirement in the RS must have a corresponding entry in the RTM.

### 14.3 Verification Types

| Type | Method | Evidence |
|------|--------|----------|
| Unit Test | Automated test | Qualified commit hash (CI-verified) |
| Qualitative Proof | Design review | Document reference or narrative |
| Verification Record | Structured behavioral test | VR document |

### 14.4 Qualification as Event

Qualification occurs at a specific point in time when RS and RTM are both EFFECTIVE and CI passes at the qualified commit. The RTM captures this event. RS and RTM must both reach EFFECTIVE before a merge PR is created.

### 14.5 CR Closure Prerequisites (Code CRs)

A code CR cannot close until:
1. RS is EFFECTIVE with all new/modified requirements
2. RTM is EFFECTIVE with verification evidence for all requirements
3. Execution branch is merged to main
4. Merge commit is verified in post-review
