Part I

The Anatomy of Flow State

Flow State is two engines bolted to one document. A geometric constraint solver (CAD) on one side, a particle simulator (physics) on the other, with a Compiler that translates between them and a Scene that conducts the whole thing frame by frame. The brilliance — and most of the bugs — live in how those pieces talk.

01Two Domains, One Document

The whole codebase is built around a single discipline: never let CAD geometry and physics particles share state. They communicate only through a one-direction-at-a-time bridge.

The Compiler is bidirectional. Static atoms (is_static=1) are passively teleported by their parent geometry. Tethered atoms (is_static=3) push back via spring forces, making CAD entities responsive to physics.

The compressed taxonomy

Sketch
The CAD document. Vector geometry plus logical constraints. Knows nothing about particles.
Simulation
Particle physics. Knows nothing about lines or constraints — just flat arrays.
Compiler
Reads Sketch, writes Simulation. Two-way coupling for tethered atoms (forces flow back).
Scene
The conductor. Owns Sketch + Sim + Compiler. Implements the 8-step update loop.
Session
Transient view state — camera, selection, active tool, focused widget.
Tool
Mouse-event handler for one operation. Talks to the world only through ToolContext.
ToolContext
The narrow facade between tools and the rest of the app. The Air Gap, made of Python.
Command
The atomic unit of mutation. Every change to the Vault flows through one.

02The Air Gap

A foundational rule: the UI cannot mutate the Data Model directly. Every change must flow through a Command, recorded in the history. Replayability and undo depend on it.

Forbidden direct mutation versus the conforming Command path. The Command path adds three concrete benefits: an undo entry, a redo entry, and a deterministic replay if state is reconstructed from history.
Why the rule is absolute

Physics is irreversible. The CAD solver introduces effects (rotation, constraint propagation) that aren't simple deltas. Without a recorded Command, undo can't restore the pre-mutation state. Even worse, replay (loading a session and re-applying its history) silently diverges.

The one documented exception

Brush operations on particles (Scene.paint_particles / Scene.erase_particles) bypass the Command queue and snapshot-undo themselves. Particles are too cheap and too volatile to wrap each splat in its own command. Snapshots are taken on mousedown (ctx.snapshot_particles()) so undo still works for the brush stroke as a whole.

Air Gap, in one sentence

Tools propose changes; the Scene records them. Anything that skips that handshake is a critical defect — including, historically, the SourceTool crash that triggered CR-112.

04Scene: The Orchestrator

Scene is the single document. It owns the CAD state, the physics state, the bridge between them, and the undo stack. Everything else — UI, tools, the input handler — exists to feed it events.

What Scene owns

class Scene:
    def __init__(self, skip_warmup=False):
        # CAD Domain
        self.sketch          = Sketch()
        self.material_manager = MaterialManager(self.sketch)
        self.geo             = GeometryManager(self.sketch)

        # Physics Domain (no CAD knowledge)
        self.simulation      = Simulation(skip_warmup=skip_warmup)

        # Bridge — knows both sides
        self.compiler        = Compiler(self.sketch, self.simulation, self.material_manager)

        # Particle brush (uses the snapshot-undo exception)
        self._brush          = ParticleBrush(self.simulation)

        # History — the source of truth for "what happened"
        self.commands        = CommandQueue(max_history=50)

        # Boundary conditions
        self.process_objects = []

        # Dirty flags govern the orchestration loop
        self._topology_dirty = False
        self._geometry_dirty = False

The two dirty flags

Performance hinges on knowing whether a change altered topology (atom count) or just geometry (atom positions). The Scene tracks both via command.changes_topology:

FlagSet byTriggersCost
_topology_dirty AddLine, AddCircle, RemoveEntity, AddConstraint, SetMaterial, SetEntityDynamic compiler.rebuild() — full atom regeneration + joint-id assignment + Numba warmup if needed Expensive (O(entities × atoms))
_geometry_dirty MoveEntity, SetPoint, SetCircleRadius, SetEntityGeometry, ToggleAnchor simulation.sync_entity_arrays() + sync_static_atoms_to_geometry() — flat array copy Cheap (O(entities))

This split is what lets dragging an endpoint feel responsive. The drag fires geometry-only commands at 60 Hz — no rebuilds, no kernel warmups, just array writes.

05The 8-Step Update Loop

Scene.update(dt, geo_time, run_physics, physics_steps) is the single frame entry point. The order matters — the loop is engineered for two-way CAD↔Physics coupling, and shuffling steps breaks invariants the rest of the code depends on.

  1. Topology Check
    If _topology_dirty, call self.rebuild() — full Compiler.rebuild() + snap tethered atoms to anchors. Clear both dirty flags.
    scene.py:293-296
  2. Driver Update
    Snapshot constraint values, call sketch.update_drivers(geo_time) to animate any time-driven constraints (e.g., motors), then snapshot again to detect change. geometry_moved = True if anything changed.
    scene.py:298-305
  3. Sync Entity Arrays (pre-physics)
    If geometry is dirty, copy current entity positions into Simulation flat arrays via sync_entity_arrays() and teleport static atoms via sync_static_atoms_to_geometry(). This is the fast path — no recompile.
    scene.py:307-314
  4. Physics Sub-Loop (Two-Way Coupling)
    Clear entity force accumulators, run simulation.apply_tether_forces() (Numba kernel), retrieve accumulated forces with get_entity_forces(), apply force + torque to each dynamic entity, integrate, then sync back. This is where particles push back on geometry.
    scene.py:316-320 → _run_physics_coupling: 354-415
  5. Constraint Solve
    If sketch.constraints exist or sketch.interaction_data is set (User Servo active), run sketch.solve(). PBD iterations. The interaction data is treated as the highest-priority constraint.
    scene.py:323-326
  6. Post-Constraint Sync
    After the solver moves geometry, re-sync entity arrays and re-teleport static atoms so the next physics step sees the corrected positions.
    scene.py:328-333
  7. Process Objects
    For each enabled ProcessObject in self.process_objects, call obj.execute(simulation, dt). Sources spawn particles; sinks absorb. Particles spawned this frame participate in the next frame's physics.
    scene.py:335-339
  8. Physics Step
    If physics is enabled and not paused, call simulation.step(physics_steps) — Verlet integration with spatial-hash collision over physics_steps sub-steps.
    scene.py:341-343
Why "two-way" matters

If you've worked with one-way CAD-to-physics pipelines, the loop above might look elaborate. The two extra sync passes (steps 3 and 6) and the dedicated physics sub-loop (step 4) exist specifically because tethered atoms accumulate forces that flow back into entity dynamics. That's how a CAD-drawn flexible membrane can deform under particle pressure — the same line you drew with the LineTool can be pushed around by hot fluid hitting it.

What does not happen during update

  • No event handling. Events are dispatched by InputHandler before update() is called.
  • No rendering. FlowStateApp.render() is a separate phase after update().
  • No autosave / file I/O. Save/load happens through dialogs, not the loop.

06The Command Pattern

Every change to the Vault is a Command. Three orthogonal flags govern its behavior: historize (does it enter the undo stack?), supersede (does it replace the previous command?), and changes_topology (does it require a rebuild?). Mastering these three flags is mastering 80% of the command system.

The contract

class Command(ABC):
    changes_topology = False          # class-level — override per type

    def __init__(self, historize=True, supersede=False):
        self.historize = historize # add to undo stack?
        self.supersede = supersede # replace previous?

    @abstractmethod
    def execute(self) -> bool: ...
    @abstractmethod
    def undo(self): ...
    def redo(self): return self.execute()
    def merge(self, other) -> bool: return False

Three flags, six interesting combinations

historizesupersedePatternUsed byEffect
✓✗ Initial mutation MOUSEDOWN of a creation tool Push new entry on the stack. Clear the redo branch.
✓✓ Live preview update MOUSEMOVE of LineTool, SelectTool drag Pop+undo previous, execute new, push. Stack height stays constant. Final MOUSEUP version stays.
✗✗ Transient ephemeral Rare — used when feedback should not be undoable at all Execute, do not record, do not clear redo.
✓✗ + merge() Mergeable continuation Sequential drags of the same handle If top-of-stack accepts a merge, combine; otherwise push new.

Lifecycle of a single line draw

The supersede trick: every MOUSEMOVE replaces the in-flight command on the stack. Net effect after MOUSEUP: a single, undoable "you drew a line from A to B" entry — not 47 micro-edits.

Where the commands live

FileCommandschanges_topology?
model/commands/geometry.py AddLineCommand, AddCircleCommand, AddRectangleCommand, RemoveEntityCommand, SetEntityGeometryCommand, MoveEntityCommand, MoveMultipleCommand Adds/removes ✓ · moves ✗
model/commands/points.py SetPointCommand, SetCircleRadiusCommand ✗ (geometry only)
model/commands/constraints.py AddConstraintCommand, RemoveConstraintCommand, ToggleAnchorCommand, ToggleInfiniteCommand, SetDriverCommand Add/remove ✓ · toggle ✗
model/commands/properties.py SetPhysicalCommand, SetMaterialCommand, SetEntityDynamicCommand ✓ (changes atomization or tethering)
model/commands/composite.py CompositeCommand (groups multiple commands as one undoable unit) Inherits from children
discard() vs undo() — a subtle distinction

scene.undo() moves the top command to the redo stack — you can press Ctrl+Y to bring it back. scene.discard() drops it permanently. Tools call discard() when the user explicitly cancels (Escape during a draw); they call undo() when the user presses Ctrl+Z. Same internal mechanism, different intent — and the redo stack is the difference.

07Sketch & The Solver

The Sketch is the CAD half of the document. It holds entities, constraints, and one transient buffer that turns mouse drags into solver constraints — the User Servo.

Entities & constraints

Three primitive entity types in model/geometry.py:

EntityPointsGeometryNotable flags
Point1(x, y) anchoris_handle — marks ProcessObject handles
Line2start, endis_reference — non-physical guide; dynamic — tethered or static
Circle1center, radiusis_reference, dynamic

Constraints come in two flavors that the solver treats differently:

ClassExamplesSolver phase
Unary (single-entity) LENGTH, RADIUS, HORIZONTAL, VERTICAL, FIX Phase 2 — establishes "entity truth"
Binary (multi-entity) COINCIDENT, COLLINEAR, PARALLEL, PERPENDICULAR, EQUAL, TANGENT, DISTANCE Phase 3 — propagates between entities

The User Servo

The Sketch.interaction_data field is the cleverest piece of the whole CAD architecture. It is not a Constraint object — it's a transient dict that the solver reads as highest-priority constraint at the start of each iteration:

sketch.interaction_data = {
    'target':     (x, y),       # where the mouse is in world coords
    'entity_idx': int,         # which entity is being dragged
    'handle_t':   float | None, # body drag: parametric position 0.0–1.0 on a line
    'point_idx':  int   | None, # endpoint drag: which control point
}

The solver phases for each iteration:

for _ in range(iterations):
    # Phase 1: User Servo (the mouse "pulls" the entity)
    if interaction_data:
        Solver._solve_interaction(entities, interaction_data)
    # Phase 2: Unary constraints (LENGTH, RADIUS, ...)
    for c in unary_constraints:
        Solver.solve_single(c, entities)
    # Phase 3: Binary constraints (COINCIDENT, PARALLEL, ...)
    for c in binary_constraints:
        Solver.solve_single(c, entities)

This phasing is what makes a constrained drag feel physical: pulling on a length-locked line endpoint produces rotation around the other anchor, not stretching.

Two backends, one API

Legacy (Python OOP)

Per-constraint methods: _solve_coincident_pt_pt, _solve_length, etc. Easy to step through with a debugger, easy to extend with a new constraint type. Slow — Python loops over every iteration.

Numba (JIT kernels)

Constraints flattened into typed numpy arrays; kernel functions in model/solver_kernels.py operate on flat data. Order-of-magnitude faster. JIT-compiled at first use; warmup amortized at scene load (skipped in Editor mode for fast startup).

Toggle live with F9. Iteration count adjustable with F10/F11. Both backends produce identical results; the toggle exists for benchmarking.

08The Compiler Bridge

The Compiler reads CAD geometry and writes particle data. It produces two distinct kinds of atoms — and the difference between them is the difference between a static wall and a dynamic, force-coupled membrane.

Static atoms vs tethered atoms

Both atom kinds carry the same tether metadata (tether_entity_idx, tether_local_pos). The difference is tether_stiffness: zero on static atoms (just teleport), non-zero on tethered atoms (Hooke's law spring with force feedback into entity dynamics).

Three Compiler entry points

MethodWhen calledCostWhat it does
compiler.rebuild() Topology change (entity added/removed, material change, dynamic toggle) Expensive (O(entities × atoms)) Compact arrays, clear joint_ids, iterate entities, generate atoms via _compile_line() / _compile_circle(), assign joint IDs via Union-Find, sync entity arrays.
simulation.sync_entity_arrays(entities) Position change (every frame if _geometry_dirty) Cheap (O(entities)) Copy current entity positions into the flat per-entity arrays the kernels read.
simulation.sync_static_atoms_to_geometry() Same trigger as above Cheap (O(static atoms)) Re-interpolate each static atom's position from its parent entity's current geometry using the stored local_t parameter.

Joint IDs: the coincidence-aware optimization

When two lines share a coincident endpoint, the atoms at that endpoint would normally repel each other through Lennard-Jones forces. The Compiler runs a Union-Find pass after compilation: atoms at coincident vertices receive a shared, non-zero joint_id. The physics kernel skips LJ calculations between atoms with the same joint_id. Result: clean joints, no exploding corners.

09Simulation: Pure Physics

engine/simulation.py contains no CAD vocabulary. It speaks only particle — flat numpy arrays, a Verlet integrator, a spatial hash, a Lennard-Jones kernel. Everything CAD-shaped is the Compiler's responsibility.

The arrays

ArrayShapeMeaning
pos_x, pos_y(N,)Particle positions
vel_x, vel_y(N,)Particle velocities (Verlet uses position deltas internally)
is_static(N,) int0 = fluid, 1 = wall, 3 = tethered to entity
sigma, epsilon, mass(N,)Per-particle Lennard-Jones parameters
color(N, 3)RGB for rendering
joint_id(N,) int0 = free; non-zero = belongs to a coincidence cluster
tether_entity_idx(N,) intParent entity index (-1 if none)
tether_local_pos(N, 2)Stored local-coordinate of tether anchor
tether_stiffness(N,)Spring constant; 0 for static atoms
entity_pos, entity_vel(E, 2)Per-entity rigid-body state, synced from Sketch each frame
entity_forces(E, 3)Force_x, force_y, torque accumulators (cleared per frame)

What simulation.step() does

  1. Apply external forces (gravity, etc.) to fluid particles
  2. Verlet integration: pos += vel * dt + 0.5 * acc * dt²
  3. Build spatial hash by binning particles into a coarse grid
  4. Lennard-Jones force computation using the spatial hash for neighbor lookup
  5. Skip force calculation for atoms sharing the same non-zero joint_id
  6. Update velocities
  7. (For tethered atoms) accumulate forces back into entity_forces

All five kernels are Numba-compiled. The first call after import is slow (warmup) — subsequent calls run at near-C speed. Warmup is preloaded at scene init unless skip_warmup=True (Editor mode bypasses physics entirely so warmup is wasted there).

11Tools & The User Servo in Practice

Every tool is a thin event handler that constructs Commands. The Tool base class has only self.ctx and self.name — no self.app backdoor. Subclasses split into a "creation" lineage (GeometryTool) and standalone tools (BrushTool, SelectTool, SourceTool).

Tool taxonomy

All tools inherit Tool(ctx, name). GeometryTool and SelectTool add the two @property sketch / @property scene accessors that delegate to ctx._get_sketch() / ctx._get_scene() — the very pattern Part II is about.

An idealized handle_event

class LineTool(GeometryTool):
    def handle_event(self, event, layout):
        if event.type == pygame.MOUSEBUTTONDOWN:
            world = self.ctx.screen_to_world(*event.pos)
            self.start = world
            self.ctx.execute(AddLineCommand(self.sketch, world, world, historize=True))
            self.ctx.set_interaction_data(world, entity_idx=self._last_idx(), point_idx=1)
        elif event.type == pygame.MOUSEMOTION and self.dragging:
            world = self.ctx.screen_to_world(*event.pos)
            self.ctx.execute(AddLineCommand(self.sketch, self.start, world, supersede=True))
            self.ctx.update_interaction_target(world)
        elif event.type == pygame.MOUSEBUTTONUP:
            self.ctx.clear_interaction_data()
            self.dragging = False

Note the dependency: the tool needs self.sketch to construct the AddLineCommand. That's where ctx._get_sketch() earns its keep — and that's the source of Part II's discomfort.

12Input Dispatch & UI

Pygame events run through a 4-layer chain. Each layer can consume the event, blocking lower layers. Modal precedes Global so dialogs can capture keys before global hotkeys fire.

The cascade: System catches OS-level events, Modal grabs anything when a dialog is up, Global owns hotkeys, HUD covers everything else (and is where the live tool lives).

The Modal Stack

AppController maintains a list-based stack via push_modal() / pop_modal(). Both methods cascade reset_interaction_state() through the entire UI tree — preventing "ghost click" bugs where a button registered as hovered before a modal opened still appears clickable when the modal closes.

The Overlay Protocol

Widgets that need to spawn floating elements (dropdowns, tooltips) implement OverlayProvider. The UIManager iterates all registered providers and renders their overlays after the main tree, ensuring correct Z-ordering. The rule: never hardcode widget-specific overlay calls in UIManager. Use the protocol.

13Process Objects

Process objects are the boundary conditions of the simulation: things that emit, absorb, heat, or measure particles. Each one owns a set of "handle" Points — real Sketch entities marked is_handle=True — that can participate in CAD constraints just like any other point.

The protocol

class ProcessObject(ABC):
    handles: Dict[str, Point]   # named handle Points (e.g. 'center')
    enabled: bool

    @abstractmethod
    def execute(self, simulation, dt): ...        # called in step 7

    @abstractmethod
    def get_geometry_for_rendering(self): ...   # dashed circles, etc.

    def register_handles(self, sketch): ...        # add handle Points to sketch.entities
    def unregister_handles(self, sketch): ...      # remove on deletion
    def to_dict(self): ...                          # save
    @staticmethod
    def from_dict(data): ...                       # load

Source — the canonical implementation

A Source spawns particles inside a circular region at a configured rate. Each frame:

  1. Accumulate spawn credit: _spawn_accumulator += rate * dt
  2. Compute an adaptive attempt count from a rolling success-rate average
  3. Rejection-sample positions inside the circle; check existing-particle proximity
  4. Spawn until the accumulator drops below 1.0 or the per-frame cap (50) is hit

The center handle is a real Point in the Sketch — drag it, anchor it, constrain it to a line. The Source moves with it. That's the elegance: process objects don't have a separate transform system; they ride on Sketch entities.

14State Classification Recap

Every piece of state in Flow State falls into one of two buckets. Knowing which bucket a piece of state belongs to determines whether you mutate it through a Command or directly. This is the most important rule in the codebase.

The Vault — Persistent State

Must use Commands.

Data that constitutes "the document." If the user saved their work right now, this is what would be in the file.

  • Entity coordinates (Line.start, Circle.radius, etc.)
  • Constraints and their values
  • Materials
  • Anchored / dynamic flags
  • Process object configuration
  • Particle simulation state (with the brush exception)

Rule: Zero trust. No direct mutation. Through scene.execute() only.

The Lobby — Transient State

Direct access OK.

State that describes how the user is looking at the document. None of this would be in a saved file (other than view restoration on load).

  • Camera zoom, pan
  • Selection sets (walls, points)
  • Active tool
  • Focused widget
  • Modal stack
  • Status bar message
  • Snap target preview

Rule: Direct mutation via Managers is acceptable.

The asymmetry: interaction_data

The User Servo's drag-target buffer is transient state — there's no reason to undo a mouse position — but it lives on Sketch, in the Vault, rather than on Session, in the Lobby. CR-112 enforced facade-only writes (ctx.set_interaction_data) which restored discipline at the tool boundary, but the underlying field still lives where it always has. This is a documented residual asymmetry — recorded as REQ-FS-ARCH-003 in the RTM.

Part II

The Tool-Facade Follow-Up

On the CR-112 cycle 4 RTM review, both tu_ui and tu_sketch independently flagged the same architectural concern. It wasn't a blocker — they both labeled it as a backlog candidate — but it's worth understanding why two independent reviewers reached the same conclusion, and what the available paths forward are.

15Where We Are

CR-112 made tools talk only to ToolContext. But two of ToolContext's 30+ methods are leading-underscore "internal," and they're load-bearing. The Air Gap holds because of a convention, not a wall.

The reviewer observation, restated

What both TUs noticed

The leading underscore on _get_sketch() / _get_scene() says "internal — don't call from outside." But the documented Tool contract relies on tools calling them — that's how subclasses build commands. The convention says "don't"; the reality says "but you must."

The Air Gap is enforced by discipline (a code review will catch a misuse) rather than by structure (a type system or import boundary that physically prevents misuse).

The Air Gap holds today because every tool author is reviewed. Fine for a single-developer codebase. As the project grows — or as agents start authoring tools — discipline-only enforcement is the cheap link in the chain.

16Three Candidate Paths

The TU reviewers sketched three ways to structurally enforce what's currently held by convention. Each one trades off differently — and they're not mutually exclusive.

Path A — Rename to first-class facade methods

Drop the underscore. Promote _get_sketch / _get_scene to public methods, with WARNING docstrings making the "construct commands; don't mutate" intent explicit. The pattern already exists in the codebase: ctx.get_entity_direct() uses exactly this approach.

class ToolContext:
    # ...
    def get_sketch(self):
        """
        Get sketch reference for command construction.

        WARNING: This provides direct model access. Use ONLY for:
          - Constructing Command instances that need a Sketch parameter
          - Read-only queries

        Do NOT mutate sketch.entities or sketch.constraints directly.
        Use ctx.execute(SomeCommand(...)) for any change.
        """
        return self._app.scene.sketch

    def get_scene(self):
        """WARNING: Direct scene access. Use sparingly. ..."""
        return self._app.scene

Tool subclasses become a one-line property change:

class GeometryTool(Tool):
    @property
    def sketch(self): return self.ctx.get_sketch()
Pros
  • Tiny diff. ~10 line changes total. Could land in a single commit.
  • Self-consistent with get_entity_direct(), which already follows this pattern.
  • Honesty: docstring says "use carefully," code says "use carefully," no contradiction.
  • Doesn't constrain future architectural moves (Path B or C still possible later).
Cons
  • Still discipline-only. The WARNING docstring is no harder to ignore than the underscore is.
  • Trades one convention for another — but at least it's an honest convention.
  • Doesn't address the duplicated @property pair across two unrelated tool classes.

Path B — Introduce a typed read-only SketchView protocol

Define a Protocol (PEP 544) that exposes only the read methods commands need to access. Tools receive a SketchView, not a Sketch — the type system structurally prevents mutation.

# model/protocols.py
from typing import Protocol, runtime_checkable

@runtime_checkable
class SketchView(Protocol):
    """Read-only view of a Sketch for use in tools and command construction."""
    @property
    def entities(self) -> Sequence[Entity]: ...     # Sequence, not list
    @property
    def constraints(self) -> Sequence[Constraint]: ...
    def find_entity_at(self, x: float, y: float, r: float) -> int: ...
    # ... only the methods commands actually need ...

# tool_context.py
class ToolContext:
    def sketch_view(self) -> SketchView:
        return self._app.scene.sketch    # Sketch satisfies SketchView structurally

Commands accept SketchView as their sketch parameter. Type checkers (mypy, pyright) reject any tool that tries to call sketch.add_entity() on a SketchView.

Pros
  • Structural enforcement. Type checker refuses non-conforming code at edit time.
  • Self-documenting: the type signature says exactly what's allowed.
  • Forces an explicit decision about the tool surface — no accidental over-exposure.
  • Future-proof: adding a new method to SketchView is a deliberate review event.
Cons
  • Largest change. Must update every Command class to accept SketchView in its signature.
  • Requires running a type checker in CI to actually enforce — Flow State doesn't have one yet.
  • Sequence types are immutable interfaces, but Python doesn't structurally prevent calling Sketch's mutator methods through a SketchView reference at runtime — you only get warnings at type-check time.
  • Adds a new abstraction layer that all future contributors have to learn.

Path C — Eliminate the subclass property accessors entirely

Move command construction into ToolContext. Tools never get a Sketch reference because they never need one. The facade builds the command from the tool's request and a sketch reference it already has internally.

# tool_context.py — many new factory methods
class ToolContext:
    def create_add_line_command(self, start, end, *, supersede=False):
        return AddLineCommand(self._app.scene.sketch, start, end, supersede=supersede)

    def create_move_entity_command(self, idx, dx, dy, *, supersede=False):
        return MoveEntityCommand(self._app.scene.sketch, idx, dx, dy, supersede=supersede)

    # ... one factory per command type, ~15 methods ...

# tools.py — no more @property sketch / @property scene
class LineTool(GeometryTool):    # GeometryTool loses both properties
    def handle_event(self, event, layout):
        # ...
        cmd = self.ctx.create_add_line_command(start, end, supersede=True)
        self.ctx.execute(cmd)

The create_coincident_command() that already exists on ToolContext is the prototype. Path C generalizes that pattern across all command types.

Pros
  • Strongest structural Air Gap. Tools physically cannot get a Sketch reference, ever.
  • The duplicated subclass property pair vanishes from every tool.
  • Command imports — currently in tools.py — concentrate in ToolContext, where they belong.
  • SourceTool's two direct _get_sketch() reaches become facade calls naturally.
Cons
  • Largest API surface on ToolContext: ~15 new factory methods (one per command type).
  • Each new command type now requires a corresponding ToolContext factory — extra ceremony.
  • Doesn't help with read paths inside tools (e.g., utils.get_snapped_pos(sketch.entities, ...)) — those still need some way to access entities, perhaps via iter_entities.
  • Trades one source of friction (discipline) for another (verbosity).

17Trade-Off Matrix

Same axes, three paths. The right answer depends on how much weight you put on structural enforcement versus implementation cost.

The hidden criterion

None of these axes captures the deeper question, which is: who will be writing tools next?

  • If the answer is "the same one or two humans who've been writing them," Path A buys honesty for almost no cost.
  • If the answer is "agents authoring new tools without human review," Path C is the only one that survives — discipline doesn't generalize across non-human contributors.
  • If the answer is "we want the type system to be load-bearing in this codebase generally," Path B is a strategic investment that pays off across many subsystems, not just this one.

18A Recommendation

My take, from sitting with the code and the reviewers' notes.

Recommendation

Land Path A now as a tiny bookkeeping CR. Defer Path C until you actually have a use case that justifies its weight — most likely the first time an agent (or a new contributor) writes a tool without close review.

The reasoning

Path A is honest about what already exists. The Air Gap is discipline-enforced today; calling those methods get_sketch() and get_scene() with proper warning docstrings doesn't weaken any guarantee — it just stops pretending they're internal when they're documented as the path. The lie is more dangerous than the convention.

Path C is the architecturally correct end state, but it's the kind of refactor that earns its weight only when there's a concrete pressure for it. Right now, two humans write the tools and review each other; the structural enforcement Path C provides is overkill. Adding 15 factory methods to ToolContext for two developers who can be trusted to call execute(SetEntityGeometryCommand(sketch, ...)) correctly is process for its own sake.

Path B is the most interesting strategically but the most expensive tactically. Flow State doesn't run mypy in CI today — adopting Path B means adopting that infrastructure too. That's a decision that should be made on its own merits, not as a side effect of cleaning up two property accessors.

Trigger conditions for revisiting

Specific signals that would push toward Path C:

  • An agent writes a tool that violates the Air Gap (a CR-2026-001-style incident, repeating).
  • A new contributor — human or otherwise — uses _get_sketch() for direct mutation and ships it.
  • Tools start being authored from outside the core team (mods, plugins, user scripts).
  • The tool surface stabilizes such that the 15 factory methods become a closed, well-understood set.

Until any of those happens, the cost-benefit on Path C doesn't pencil out. The reviewers were right to flag it, and right to label it a backlog candidate. Land A; queue C; revisit when there's a forcing function.
